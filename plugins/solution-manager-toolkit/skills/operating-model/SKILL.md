# Target Operating Model

Domain knowledge for defining how roles, processes, and responsibilities change alongside systems.

## When to Use

- Defining what changes for people when a system launches
- Reviewing a solution that has technical artifacts but no people/process plan
- Workshop prep for stakeholder alignment on operational changes

## Components

1. **Current state** — How people work today, role by role
2. **Future state** — How people work after deployment
3. **What changes for PEOPLE** — Role by role: eliminated tasks, new responsibilities, training needs, reassignments
4. **New processes needed** — Monitoring, escalation paths, review cadences, exception handling

## Key Principles

- Technology is only half the solution. If people don't change how they work, the old process runs alongside the new system.
- Every role affected needs explicit callout — don't assume people will figure it out.
- Training needs are requirements, not afterthoughts.
- If a role is eliminated, say so plainly and note reassignment plans.
- Flag when a solution has technical artifacts but no operating model: "You've defined the technical solution but not how people and processes change."
